# PMEGP Update Pack Signing — Ops Runbook

This document is the operational runbook for **generating Ed25519 signing keys,
signing Knowledge Package update manifests, publishing packs to the CDN, and
rotating keys** for the PMEGP Assistant Android app.

It is the canonical reference for the Update Engine's trust model. The engine
implementation lives in [`src/engines/update-engine/index.ts`](../src/engines/update-engine/index.ts);
the signing tooling lives in [`scripts/`](../scripts/).

---

## 1. Trust model — short version

1. Each Knowledge Package update is shipped as **two files** on a CDN:
   - `manifest.json` — metadata + Ed25519 signature
   - `pack.json` — the actual update payload (`{ knowledge_version, data? }`)
2. The manifest is **Ed25519-signed** by a private key held only by the release
   engineer. The corresponding public key is **bundled into the APK** at
   `src/engines/update-engine/public-key.pem`.
3. On the device, the Update Engine:
   1. Fetches the manifest from a trusted URL.
   2. Recomputes the canonical JSON of the manifest (sorted keys, `signature`
      field stripped) and verifies the signature against the bundled public key
      using `@noble/ed25519`'s `verifyAsync(signature, message, publicKey)`.
   3. Downloads the pack, checks its SHA-256 against `manifest.files[0].sha256`.
   4. Parses the pack and upserts `knowledge_version` + every `data` entry into
      the `app_meta` SQLite table inside a single transaction. Existing DPRs
      (rows in `projects`) are **never** rewritten — they keep the
      `knowledgeVersion` under which they were generated.
4. If **any** step fails (bad signature, hash mismatch, malformed pack, DB
   error), the transaction rolls back and the device's knowledge version is
   unchanged. The engine **fails closed** — a bad pack is never partially
   applied.

The private key never leaves the release engineer's machine (or CI signing
host). The public key ships in every APK; rotating it requires shipping a new
APK release (see [§5 Key rotation](#5-key-rotation)).

## 3. Generating a fresh Ed25519 keypair

> The current keypair was generated by Task 6-A. You only need this section to
> rotate keys (see [§5](#5-key-rotation)) or bootstrap a new release
> environment.

Run this from the project root with `bun` (any Node ≥ 12 also works):

```bash
mkdir -p .secrets && bun -e '
const crypto = require("crypto");
const { publicKey, privateKey } = crypto.generateKeyPairSync("ed25519");
const pubPem  = publicKey.export({ type: "spki",  format: "pem" });
const privPem = privateKey.export({ type: "pkcs8", format: "pem" });
require("fs").writeFileSync("src/engines/update-engine/public-key.pem", pubPem);
require("fs").writeFileSync(".secrets/update-signing-private-key.pem", privPem);
console.log("Public key  -> src/engines/update-engine/public-key.pem");
console.log("Private key -> .secrets/update-signing-private-key.pem (gitignored)");
'
```

**Sanity-check the pair** (signs + verifies + is accepted by `@noble/ed25519`
in the argument order the engine uses):

```bash
bun -e '
const crypto = require("crypto");
const fs = require("fs");
const { verifyAsync } = require("@noble/ed25519");
const pubPem  = fs.readFileSync("src/engines/update-engine/public-key.pem", "utf8");
const privPem = fs.readFileSync(".secrets/update-signing-private-key.pem", "utf8");
const pub  = crypto.createPublicKey(pubPem);
const priv = crypto.createPrivateKey(privPem);
const msg = Buffer.from("test message");
const sig = crypto.sign(null, msg, priv);
const okNoble = await verifyAsync(sig, msg, Buffer.from(pubPem.replace(/-----.*-----/g,"").replace(/\s+/g,""), "base64").subarray(12));
const okNode  = crypto.verify(null, msg, pub, sig);
console.log({ okNode, okNoble, sigLen: sig.length });
'
```

Expect `{ okNode: true, okNoble: true, sigLen: 64 }`.

**Backup the private key.** It is not in git. Lose it and you cannot sign
updates that verify against the already-shipped public key — you would have to
rotate (new APK release) instead. Recommended: store the PEM in your team
secret manager (1Password / Vault / AWS Secrets Manager) under a name like
`pmegp/update-signing/private-key-v1`.

## 4. Signing + publishing a Knowledge Pack update

### 4.1 Build the pack payload

Author the pack as a JSON object matching the engine's `PackPayload` shape:

```json
{
  "knowledge_version": "2026.02.01",
  "data": {
    "last_knowledge_refresh": "2026-02-01T00:00:00.000Z",
    "nic_code_count": 1310,
    "machinery_catalog_count": 612
  }
}
```

- `knowledge_version` is a free-form string. Convention: `YYYY.MM.DD[-suffix]`.
  The engine treats it as opaque — any change means "new version available".
- `data` is optional. Every entry is upserted into `app_meta`
  (`key` → JSON-serialized `value`). Non-string values are JSON-serialized
  automatically by the engine.

Save this as `pack.json` in your release working directory (NOT the repo — the
published pack is a release artifact, not a source file; the only repo-resident
pack is the test fixture).

### 4.2 Compute the pack's SHA-256

```bash
sha256sum pack.json
# → 9cf12134ee9ae7afba82c4a36ec461bf0c5f660c7844e32da2e9c6cc57a24b06  pack.json
```

### 4.3 Author the unsigned manifest

Create `manifest.json` (do NOT include a `signature` field yet — the signer
will add it):

```json
{
  "version": "2026.02.01",
  "schemeCode": "PMEGP",
  "files": [
    { "path": "pack.json", "sha256": "9cf12134ee9ae7afba82c4a36ec461bf0c5f660c7844e32da2e9c6cc57a24b06" }
  ],
  "publicKeyId": "v1"
}
```

Field rules (enforced by `checkForUpdate`'s shape validation):

| Field | Type | Notes |
| --- | --- | --- |
| `version` | `string` | Must equal `pack.knowledge_version` (the engine writes the latter to the DB, but `version` is what `checkForUpdate` compares against the current knowledge version). |
| `schemeCode` | `"PMEGP"` | Literal. Rejects packs targeted at other schemes. |
| `files` | `Array<{ path, sha256 }>` | The engine's `verifyAndApply` only checks SHA-256 for **single-file packs** (`files.length === 1`). For multi-file packs, the integrity check is skipped (only signature is verified). |
| `publicKeyId` | `string` | Opaque tag identifying which public key signed this. Convention: `v1`, `v2`, … bumps on rotation. The engine does NOT look this up — it always verifies against the bundled key. It is for human/ops traceability. |

### 4.4 Sign the manifest

```bash
bun scripts/sign-manifest.mjs path/to/manifest.json
# Signed manifest: /abs/path/to/manifest.json
#   version:     2026.02.01
#   schemeCode:  PMEGP
#   publicKeyId: v1
#   signature:   <base64...>
```

This **writes the signature back into `manifest.json`'s `signature` field** in
place, and echoes it to stdout. The script:

1. Reads the manifest JSON.
2. Strips the `signature` field (if present — idempotent re-signing works).
3. Canonicalizes the remaining object: object keys sorted recursively ascending,
   arrays in source order, no inter-token whitespace. **This must byte-match the
   canonicalizer in `src/engines/update-engine/index.ts`** or the engine's
   `verifyAsync` will reject the signature. The shared `canonicalJsonStringify`
   helper is exported from `scripts/sign-manifest.mjs` and is the single source
   of truth used by both the signer and (mirrored) by the engine.
4. Signs the UTF-8 canonical bytes with Ed25519 via Node's `crypto.sign(null,
   msg, privKey)` (64-byte raw signature, no separate hash — Ed25519 is
   pure-PureEdDSA).
5. Base64-encodes the signature and writes `manifest.signature = <base64>`.

Override the private key path with `--priv-key <path>` if you keep multiple
keys (e.g. during rotation overlap).

### 4.5 Verify locally before publishing

Round-trip sanity check using the bundled public key (the same one the device
will use):

```bash
bun -e '
const crypto = require("crypto");
const fs = require("fs");
const { verifyAsync } = require("@noble/ed25519");
const { signManifest, canonicalBytesForSignature } = require("./scripts/sign-manifest.mjs");
const m = JSON.parse(fs.readFileSync("path/to/manifest.json","utf8"));
const pubPem = fs.readFileSync("src/engines/update-engine/public-key.pem","utf8");
const pub = crypto.createPublicKey(pubPem);
const ok = crypto.verify(null, canonicalBytesForSignature(m), pub, Buffer.from(m.signature,"base64"));
console.log("verify:", ok);
'
```

### 4.6 Publish to the CDN

Upload **both** files to your CDN (e.g. Cloudflare R2 / S3 + CloudFront). The
two URLs the app needs are configured by the device's `manifestUrl` and the
pack URL — the latter is conventionally `${dirname(manifestUrl)}/${files[0].path}`
but the engine accepts the pack URL as an independent parameter from the
caller (the `downloadPack(packUrl)` step).

Recommended layout:

```
https://cdn.example.com/pmegp/updates/v1/
  2026.02.01/
    manifest.json
    pack.json
  2026.01.15-sample/
    manifest.json
    pack.json
  latest/                 ← symlink or copy of the current version
    manifest.json
    pack.json
```

The device is typically configured with `https://cdn.example.com/pmegp/updates/v1/latest/manifest.json`
so the latest manifest is always one fetch away. Old version directories remain
available for debugging / rollback forensics.

**CDN requirements:**

- HTTPS only (the engine uses `fetch`; mixed-content blocking would fail
  closed, which is safe but confusing).
- Immutable cache headers on `pack.json` (content-addressed by SHA-256 in the
  manifest) — `Cache-Control: public, max-age=31536000, immutable`.
- Short TTL on `manifest.json` so version bumps propagate fast —
  `Cache-Control: public, max-age=300` (5 min) is a good default.
- CORS: NOT required (the request is from the Capacitor WebView's native HTTP
  stack on Android, not browser fetch from another origin). If you also test
  in a browser, add `Access-Control-Allow-Origin: *` on both files.

### 4.7 End-to-end smoke test

After publishing, fetch the manifest URL from a clean environment and run the
sign-manifest.mjs verifier against the fetched bytes to confirm the CDN is
serving exactly what you signed (CDN rewriting / Brotli / minification would
break the signature):

```bash
curl -s https://cdn.example.com/pmegp/updates/v1/latest/manifest.json \
  | bun -e '
    const crypto = require("crypto");
    const fs = require("fs");
    const { canonicalBytesForSignature } = require("./scripts/sign-manifest.mjs");
    let raw = ""; process.stdin.on("data",c=>raw+=c); process.stdin.on("end",()=>{
      const m = JSON.parse(raw);
      const pub = crypto.createPublicKey(fs.readFileSync("src/engines/update-engine/public-key.pem","utf8"));
      console.log("CDN manifest verifies:", crypto.verify(null, canonicalBytesForSignature(m), pub, Buffer.from(m.signature,"base64")));
    });
  '
```

Expect `CDN manifest verifies: true`.

## 5. Key rotation

Rotation is rare — Ed25519 keys don't wear out — but is required if the private
key is suspected leaked, or as part of a periodic hygiene policy.

### 5.1 Rotation flow (overlap window, zero downtime)

The device only knows the public key baked into its APK. So a rotation **must
ship a new APK** before (or alongside) signing packs with the new key. Devices
that never upgrade will keep verifying against the old key — and will refuse
new packs (fail closed, by design).

To rotate without orphaning old devices:

1. **Generate a new keypair** (see [§3](#3-generating-a-fresh-ed25519-keypair)).
   The new public key overwrites `src/engines/update-engine/public-key.pem`.
   Keep the OLD private key around (`.secrets/update-signing-private-key-v1.pem`)
   until the overlap window closes.
2. **Bump `publicKeyId`** to `v2` in new manifests.
3. **Ship a new APK** with the new `public-key.pem` bundled.
4. **Wait** for adoption to exceed your comfort threshold (e.g. 90% of
   `app_meta.last_update_check` pings report the new APK version). Devices on
   the old APK will silently fail signature verification on `v2` packs — they
   stay on their last-known knowledge version, which is safe.
5. **Destroy the old private key** (or move it to cold storage for forensic
   use). Remove `.secrets/update-signing-private-key-v1.pem`.

### 5.2 Emergency rotation (private key leak suspected)

Skip the overlap window: generate a new keypair, ship an emergency APK, and
**immediately stop signing with the old key**. Devices on old APKs will reject
all future packs (fail closed) until they upgrade — this is the intended
behavior for a leak.

### 5.3 Multi-key support (not currently implemented)

The `publicKeyId` field is the hook for future multi-key support: the engine
could maintain a `{ publicKeyId → publicKey }` map and try each. This is NOT
implemented today (the engine uses a single bundled key), so `publicKeyId` is
purely informational. Do not rely on the engine honoring it during rotation —
ship the new APK instead.

## 6. Test fixture

`scripts/create-sample-pack.mjs` generates a signed sample pack to
`tests/fixtures/sample-update-pack/`:

```bash
bun scripts/create-sample-pack.mjs
# Optional: --out <dir> --version <ver>
```

The fixture is **self-verified** by the script: after writing both files, it
re-loads the manifest, recomputes canonical bytes, and verifies the signature
against the bundled public key using Node's `crypto.verify`. If the
self-verify fails, the script exits non-zero.

To validate the fixture end-to-end against the **actual** engine libs
(`@noble/ed25519` + WebCrypto SHA-256), see the verification snippet in Task
6-A's worklog entry — it loads `manifest.json` + `pack.json`, runs the same
`verifyAsync(sig, msg, pub)` + `subtle.digest('SHA-256', packBytes)` calls the
engine makes, and asserts both match.

The fixture is regenerated from scratch (new signature, same content) when the
script is re-run; the signature is deterministic for identical inputs, so
re-running on the same machine against the same key produces byte-identical
output.

## 7. Reference: canonical JSON contract

Both signer and verifier MUST agree on the exact byte representation the
signature covers. The canonical form is defined in
`src/engines/update-engine/index.ts`'s `canonicalJsonStringify` and mirrored in
`scripts/sign-manifest.mjs`:

- `null` / primitives → `JSON.stringify(value)` (so `true`, `42`, `"hi"`,
  `null`).
- Arrays → `[` + elements (canonicalized) joined by `,` + `]` (order preserved).
- Objects → `{` + entries (key canonicalized via `JSON.stringify`, value
  canonicalized recursively) joined by `,` + `}`, with **keys sorted
  ascending lexicographically by UTF-16 code unit** (`Object.keys(obj).sort()`).
- No whitespace between any tokens.
- The `signature` field is stripped from the manifest **before**
  canonicalization (it's not part of the signed content).

If you ever need to change this contract (e.g. switch to RFC 8785 JCS), you
must update **both** the engine and the signer in the same release, and
re-sign every in-flight manifest. There is no versioning field for the
canonicalization algorithm — it's implicit.

## 8. FAQ

**Q: Can I sign on CI without exposing the private key?**
A: Yes — store the PEM as a repository secret (GitHub Actions /
GitLab CI variables) and write it to `.secrets/update-signing-private-key.pem`
in the job before invoking `bun scripts/sign-manifest.mjs`. The signer reads
from that path by default. For an explicit path, use `--priv-key <path>`.

**Q: What happens if the device is offline when an update is published?**
A: Nothing — the Update Engine only fetches when the user (or a periodic
background task) calls `checkForUpdate(manifestUrl)`. Offline = no fetch = no
update. The bundled knowledge package remains in use.

**Q: Why Ed25519 and not RSA / ECDSA P-256?**
A: Ed25519 has 64-byte signatures (vs RSA-2048's 256 bytes), is deterministic
(no nonce-reuse footgun), and is fast enough on Android's WebView for
sub-millisecond verification. `@noble/ed25519` is a pure-JS implementation
that works inside the Capacitor WebView without native bindings.

**Q: Can a single manifest cover multiple pack files?**
A: Yes — `files` is an array. But the engine's `verifyAndApply` only
**checks SHA-256 for single-file packs** (`files.length === 1`). For
multi-file packs, only the manifest signature is verified; per-file integrity
is the publisher's responsibility. If you need multi-file integrity checks,
extend `verifyAndApply` to download + hash each entry.

**Q: What if I lose the private key but no leak is suspected?**
A: You cannot sign any more packs that verify against the shipped public key.
You must rotate (new APK). The old private key is gone, but so what — you can't
sign with a key you don't have. The risk is only if someone *else* finds it.
Treat lost-key as a forced rotation: see [§5.2](#52-emergency-rotation-private-key-leak-suspected).
