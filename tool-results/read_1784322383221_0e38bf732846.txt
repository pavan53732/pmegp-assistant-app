     8→## 1. Purpose
     9→
    10→This document specifies **how the AI interview builds a project** — the one architectural gap left open by the existing documentation set.
    11→
    12→The interview is the **first touchpoint** between the beneficiary and the application. Everything downstream — the Validation Engine, Eligibility Engine, Financial Engine, DPR Engine, and PDF Engine — consumes the same object: the **Structured Project Profile**. This document defines that object, the reasoning pipeline that constructs it, the phases the interview progresses through, the role of the Knowledge Package in assisting beneficiaries who cannot answer technical questions, and the provenance and suggestion models that track data origin and verification for every field.
    13→
    14→This document sits between the user and the Validation Engine in the pipeline:
    15→
    16→```
    17→User
    18→  ↓
    19→AI Interview (this document)
    20→  ↓
    21→Structured Project Profile (Section 2)
    22→  ↓
    23→Validation Engine (doc 05 of doc 04 §5, doc 15 §2)
    24→```
    25→
    26→**No application code should be written until this document is reviewed and approved.**

    30→## 2. The Structured Project Profile (canonical schema)
    31→
    32→The Structured Project Profile is the **single data contract** between the interview layer and every downstream engine. Both the AI interview and the guided forms produce an identical instance of this object. Downstream engines consume only this object — never raw conversation messages, never partial AI outputs, never form state.
    33→
    34→The profile is a **typed, Zod-validated** object. The canonical type definition lives in `shared/types/project-profile.ts`. The shape below is the **specification** — implementation will mirror it exactly.
    35→
    36→### 2.1 Schema overview
    37→
    38→```
    39→ProjectProfile {
    40→  // ── APPLICANT ──────────────────────────────────────────────
    41→  applicant: {
    42→    name:                   string
    43→    age:                    number
    44→    gender:                 "MALE" | "FEMALE" | "OTHER"
    45→    category:               "GEN" | "SC" | "ST" | "OBC" | "MINORITY"
    46→                            | "EX_SERVICEMEN" | "PH" | "NER"
    47→    isWomen:                boolean        // derived from gender; stored for subsidy calc
    48→    phone?:                 string
    49→    email?:                 string
    50→    education:              "NONE" | "BELOW_8TH" | "8TH_PASS" | "10TH_PASS"
    51→                            | "12TH_PASS" | "GRADUATE" | "POST_GRADUATE" | "PROFESSIONAL"
    52→                            | "OTHER"
    53→    educationDetail?:       string         // free text if OTHER
    54→    entityType:             "INDIVIDUAL" | "SHG" | "TRUST" | "SOCIETY"
    55→                            | "COOP" | "PARTNERSHIP" | "LLP" | "PRIVATE_LIMITED"
    56→    entityRegistrationNo?:  string
    57→    aadhaarNo?:             string         // stored encrypted, masked in logs/prompts
    58→    panNo?:                 string
    59→    priorSubsidy:           boolean        // availed PMEGP/PMRY/other govt subsidy?
    60→    priorSubsidyDetail?:    string
    61→    edpCompleted:           boolean        // EDP training done?
    62→    edpCertificateNo?:      string
    63→    experienceYears?:       number
    64→    experienceDetail?:      string
    65→  }
    66→
    67→  // ── BUSINESS ───────────────────────────────────────────────
    68→  business: {
    69→    name:                   string         // proposed unit/project name
    70→    description:            string         // what the beneficiary wants to do, in their words
    71→    activityType:           "MANUFACTURING" | "SERVICE"  // fixed after activity resolution
    72→    nicCode:                string?        // 6-digit NIC code — fixed after activity resolution
    73→    nicDescription:         string?        // human-readable NIC label — fixed after activity resolution
    74→    sector:                 "MANUFACTURING" | "SERVICE"  // derived from NIC; fixed after activity resolution
    75→    subCategory:            "MANUFACTURING" | "SERVICE" | "TRADING" | "TRANSPORT"
    76→                                           // derived from NIC; fixed after activity resolution
    77→  }
    78→
    79→  // ── LOCATION ───────────────────────────────────────────────
    80→  location: {
    81→    state:                  string
    82→    district:               string
    83→    mandal?:                string
    84→    village?:               string
    85→    pinCode?:               string
    86→    area:                   "URBAN" | "RURAL"
    87→    premisesAddress?:       string
    88→    isHillBorderArea:       boolean        // Hill & Border Area (affects subsidy categorization)
    89→    isAspirationalDistrict: boolean        // Aspirational District
    90→    industrialAreaName?:    string
    91→    industrialAreaType?:    "INDUSTRIAL_ESTATE" | "SEZ" | "CLUSTER"
    92→                          | "DAC" | "OTHER" | null
    93→  }
    94→
    95→  // ── LAND ───────────────────────────────────────────────────
    96→  land: {
    97→    status:                 "OWN" | "RENTED" | "LEASED" | "NONE" | "FAMILY"
    98→    areaSqFt?:              number
    99→    areaSqMt?:              number
   100→    ownedLandValue?:        number         // in rupees (for project cost)
   101→    monthlyRent?:           number         // in rupees
   102→    leaseAmount?:           number
   103→    leaseYears?:            number
   104→    buildingType?:          "OWN" | "RENTED" | "CONSTRUCT"
   105→    buildingAreaSqFt?:      number
   106→    constructionCost?:      number         // in rupees (for project cost)
   107→  }
   108→
   109→  // ── CAPACITY & PRODUCTION ──────────────────────────────────
   110→  capacity: {
   111→    installedCapacity: {
   112→      unit:                 string         // e.g., "kg/month", "units/month", "litres/day"
   113→      value:                number
   114→    }
   115→    projectedCapacityUtil:  number         // percentage, typically 60-80%
   116→    workingDaysPerMonth:    number         // typically 25-26
   117→    workingHoursPerDay:     number         // typically 8
   118→    shifts:                 number         // 1, 2, or 3
   119→  }
   120→
   121→  // ── MACHINERY & EQUIPMENT ──────────────────────────────────
   122→  machinery: {
   123→    items:                  MachineryItem[]
   124→    totalCost:              number         // in rupees — sum of items, authoritative for project cost
   125→  }
   126→
   127→  MachineryItem {
   128→    name:                   string
   129→    specification?:         string
   130→    quantity:               number
   131→    unitCost:               number         // in rupees
   132→    totalCost:              number         // quantity × unitCost
   133→    supplier?:              string
   134→    quotationRef?:          string         // links to attachment if OCR'd quotation exists
   135→    source:                 "USER" | "AI" | "KNOWLEDGE" | "OCR"
   136→  }
   137→
   138→  // ── RAW MATERIALS ──────────────────────────────────────────
   139→  rawMaterials: {
   140→    items:                  RawMaterialItem[]
   141→    totalMonthlyCost:       number         // in rupees — sum of items
   142→  }
   143→
   144→  RawMaterialItem {
   145→    name:                   string
   146→    specification?:         string
   147→    monthlyQuantity:        number
   148→    unit:                   string
   149→    unitRate:               number         // in rupees
   150→    totalMonthlyCost:       number         // quantity × unitRate
   151→    source:                 "USER" | "AI" | "KNOWLEDGE" | "OCR"
   152→  }
   153→
   154→  // ── EMPLOYEES ──────────────────────────────────────────────
   155→  employees: {
   156→    skilled: {
   157→      male:                 number
   158→      female:               number
   159→      monthlyWagePerPerson: number
   160→    }
   161→    unskilled: {
   162→      male:                 number
   163→      female:               number
   164→      monthlyWagePerPerson: number
   165→    }
   166→    administrative: {
   167→      count:                number
   168→      monthlyWagePerPerson: number
   169→    }
   170→    totalMonthlyWages:      number         // computed sum
   171→    totalEmployment:        number         // total persons
   172→  }
   173→
   174→  // ── UTILITIES & OVERHEADS ──────────────────────────────────
   175→  utilities: {
   176→    monthlyPowerCost:       number         // in rupees
   177→    monthlyWaterCost:       number
   178→    monthlyRentCost:        number
   179→    monthlyMaintenanceCost: number
   180→    monthlyTransportCost:   number
   181→    monthlyCommunicationCost: number
   182→    monthlyInsuranceCost:   number
   183→    monthlyMiscCost:        number
   184→    totalMonthlyOverheads:  number         // computed sum
   185→  }
   186→
   187→  // ── FINANCIAL ASSUMPTIONS ──────────────────────────────────
   188→  financials: {
   189→    machineryAndEquipment:  number         // in rupees — alias for machinery.totalCost
   190→    otherFixedAssets:       number         // in rupees
   191→    preOperativeExpenses:   number         // in rupees
   192→    buildingAndCivilWorks:  number         // in rupees
   193→    totalFixedCapital:      number         // computed sum of above
   194→    workingCapital:         number         // in rupees
   195→    totalProjectCost:       number         // fixedCapital + workingCapital
   196→    interestRate:           number         // annual % (user-provided or Knowledge Package default)
   197→    loanTenureYears:        number
   198→    repaymentMoratoriumMonths: number      // typically 6
   199→    collateralOffered?:     string
   200→    projectedMonthlySales:  number         // in rupees
   201→  }
   202→
   203→  // ── WORKING CAPITAL BREAKDOWN ──────────────────────────────
   204→  workingCapitalDetail: {
   205→    rawMaterialDays:        number         // typically 30-45 days
   206→    workInProgressDays:     number         // typically 7-15 days
   207→    finishedGoodsDays:      number         // typically 15-30 days
   208→    creditorsDays:          number         // typically 15-30 days
   209→    computedWorkingCapital: number?        // days-based calculation result, in rupees
   210→    method:                 "USER_PROVIDED" | "DAYS_BASED" | "PERCENTAGE_OF_PROJECT_COST"
   211→  }
   212→
   213→  // ── MARKET ─────────────────────────────────────────────────
   214→  market: {
   215→    targetMarket:           string         // description of target customers/area
   216→    marketDemand?:          string         // demand assessment
   217→    competition?:           string
   218→    marketingStrategy?:     string
   219→    sellingPricePerUnit?:   number
   220→    sellingPriceUnit?:      string
   221→  }
   222→
   223→  // ── QUOTATIONS & ATTACHMENTS ───────────────────────────────
   224→  attachments: {
   225→    items:                  AttachmentRef[]
   226→  }
   227→
   228→  AttachmentRef {
   229→    id:                     string
   230→    type:                   "QUOTATION" | "IDENTITY_PROOF" | "ADDRESS_PROOF"
   231→                            | "LAND_DOCUMENT" | "EDP_CERTIFICATE" | "OTHER"
   232→    fileName:               string
   233→    ocrExtractedFields?:    Record<string, string>   // from OCR engine
   234→    linkedMachineryIndex?:  number         // index into machinery.items[]
   235→    status:                 "PENDING_REVIEW" | "CONFIRMED" | "REJECTED"
   236→  }
   237→
   238→  // ── VALIDATION METADATA ────────────────────────────────────
   239→  validation: {
   240→    completeness:           number         // 0–100, computed by Validation Engine
   241→    missingFields:          string[]       // mandatory fields not yet filled
   242→    errors:                 ValidationError[]
   243→    contradictions:         Contradiction[]
   244→  }
   245→
   246→  ValidationError {
   247→    fieldPath:              string         // e.g., "applicant.age"
   248→    code:                   string         // e.g., "TOO_LOW", "INVALID_FORMAT"
   249→    message:                string
   250→  }
   251→
   252→  Contradiction {
   253→    fields:                 string[]       // e.g., ["land.status", "land.ownedLandValue"]
   254→    description:            string
   255→  }
   256→
   257→  // ── PROVENANCE METADATA ──────────────────────────────────
   258→  // Every field carries two orthogonal dimensions:
   259→  //   Source:      WHERE the value came from (USER, AI, OCR, KNOWLEDGE, null)
   260→  //   Verification: whether the user has explicitly confirmed it
   261→  // See Section 9 for the full provenance model.
   262→  provenance: {
   263→    perField:               Record<string, FieldProvenance>
   264→    aggregate:              number         // 0–1, weighted average of engine-ready fields
   265→  }
   266→
   267→  FieldProvenance {
   268→    source:                 "USER" | "AI" | "OCR" | "KNOWLEDGE" | null
   269→    verification:           "UNVERIFIED" | "CONFIRMED" | "VALIDATED"
   270→    confirmedAt?:           string         // ISO timestamp of user confirmation
   271→    extractConfidence?:     number         // 0–1, from AI/OCR extraction confidence
   272→    knowledgeSource?:       string         // Knowledge Package entry ref (if source = "KNOWLEDGE")
   273→  }
   274→
   275→  // ── COMPLETION METADATA ────────────────────────────────────
   276→  completion: {
   277→    currentPhase:           InterviewPhase // which interview phase is active
   278→    phaseProgress:          Record<InterviewPhase, PhaseProgress>
   279→    startedAt:              string         // ISO timestamp
   280→    lastUpdatedAt:          string         // ISO timestamp
   281→    interactionCount:       number         // number of AI or form interactions
   282→  }
   283→
   284→  PhaseProgress {
   285→    status:                 "NOT_STARTED" | "IN_PROGRESS" | "COMPLETED" | "NEEDS_REVIEW"
   286→    completedFields:        number
   287→    totalFields:            number
   288→  }
   289→}
   290→
   291→InterviewPhase =
   292→  | "APPLICANT_DISCOVERY"
   293→  | "BUSINESS_DISCOVERY"
   294→  | "ACTIVITY_RESOLUTION"
   295→  | "PROJECT_SIZING"
   296→  | "FINANCIAL_PLANNING"
   297→  | "REVIEW"
   298→  | "VALIDATION_COMPLETION"
   299→```
   300→
   301→### 2.2 Design rules for the profile
   302→
   303→1. **Every downstream engine consumes this object.** The Eligibility Engine, Financial Engine, DPR Engine, and PDF Engine all receive (a subset of) the `ProjectProfile` as their primary input. None reads conversation history, form state, or partial data.
   304→
   305→2. **Money is never stored as floating point.** All rupee amounts are integers (whole rupees or paise — decided at implementation, recorded in [05-financial-engine.md](05-financial-engine.md)). The Financial Engine consumes integer values and returns integer values.
   306→
   307→3. **The profile is the output of the interview, not the interview itself.** Conversation messages, UI state, and the question-selection logic live in `features/ai/` — they are not part of the profile.
   308→
   309→4. **Optional fields (`?`) vs mandatory fields.** The Validation Engine defines which fields are mandatory for the profile to be considered complete. The schema itself allows `?` for fields that may be unknown or inapplicable — the Validation Engine enforces the business rules.
   310→
   311→5. **The profile is immutable during a single engine run.** An engine receives a snapshot of the profile; it does not observe mid-interview mutations.
   312→
   313→6. **Both interaction models produce the same type.** There is no `AiProjectProfile` vs `FormProjectProfile`. The Zod schema that validates the AI interview output is the same Zod schema that validates the guided forms.
   314→
   315→### 2.3 Field ownership
   316→
   317→Every field or field group in the Structured Project Profile has exactly one **authoritative owner** — the subsystem that is responsible for producing or computing its value. Other subsystems may read or display the field, but they never become its author.
   318→
   319→This creates a powerful invariant: **no subsystem silently overwrites another subsystem's data.**
   320→
   321→| Owner | Fields | Notes |
   322→|-------|--------|-------|
   323→| **Applicant (user input)** | `applicant.name`, `applicant.age`, `applicant.gender`, `applicant.category`, `applicant.phone`, `applicant.email`, `applicant.education`, `applicant.entityType`, `applicant.aadhaarNo`, `applicant.panNo`, `applicant.priorSubsidy`, `applicant.address.*` | Directly provided by the beneficiary. The interview/AI may extract these from conversation but the authoritative source is the user. |
   324→| **Knowledge Engine** (after user confirmation) | `business.nicCode`, `business.nicDescription`, `business.sector`, `business.subCategory`, `business.activityType` | Looked up from the NIC code database. The user confirms the lookup result, making it authoritative. |
   325→| **Interview / AI extraction** | `business.description`, `business.name`, `business.constitution` | Extracted from the user's conversational description. The user confirms. |
   326→| **Applicant (user input)** | `location.district`, `location.state`, `location.pincode`, `location.urbanRural` | Directly provided or derived from district lookup. |
   327→| **Applicant (user input)** | `land.status`, `land.ownedLandValue`, `land.rentedMonthlyRent` | Directly provided. |
   328→| **Applicant + Knowledge Engine** | `capacity.*` | Production capacity is user-provided; unit benchmarks come from Knowledge Package as reference. |
   329→| **Applicant + Knowledge Engine** | `machinery.items[]` | User selects/suggests machinery; Knowledge Package provides reference lists and indicative costs. Each item carries its own `source` tag. |
   330→| **Applicant + Knowledge Engine** | `rawMaterials.items[]` | Same pattern as machinery. |
   331→| **Applicant (user input)** | `employees.*` | Directly provided. |
   332→| **Applicant (user input)** | `utilities.*` | Directly provided. |
   333→| **Applicant (user input)** | `financials.*` | Directly provided (project cost, promoter's contribution, loan amount). |
   334→| **Financial Engine** | *(computed results, stored in separate tables)* | Subsidy amount, DSCR, bank loan breakdown, interest calculations. These are NOT stored in the profile — they are snapshotted in `project_financials` (doc 03). The profile only stores the *inputs* to financial calculations. |
   335→| **Eligibility Engine** | *(computed results, stored in separate tables)* | Eligibility status, applicable scheme rules, maximum project cost. Stored in separate tables, not in the profile. |
   336→| **DPR Engine** | *(generated document)* | The DPR is an output that reads the profile — it does not write back to it. |
   337→| **Validation Engine** | `validation.*` | The validation metadata section (errors, warnings, contradictions, completeness) is owned and written exclusively by the Validation Engine. |
   338→| **Interview orchestration** | `provenance.*`, `completion.*` | Provenance and completion metadata are owned by the interview layer. |
   339→
   340→**Key rules:**
   341→
   342→1. **Engines never write to the profile.** The Financial Engine, Eligibility Engine, and DPR Engine are pure functions — they receive the profile and produce outputs. Their outputs are stored in their own tables, never written back into the profile.
   343→2. **The Validation Engine only writes to `validation.*`.** It reads the entire profile but only modifies the validation metadata section.
   344→3. **Derived fields have a defined owner.** `isWomen` (derived from `gender`) is owned by the Interview orchestration — it computes and writes it immediately after the gender field is set.
   345→4. **Field ownership prevents responsibility creep.** If a future developer is tempted to have the Financial Engine "fix" a machinery cost, the ownership table makes it clear that this is a violation — machinery costs are owned by the Applicant + Knowledge Engine.

   349→## 3. The AI Reasoning Pipeline
   350→
   351→Every user message during the AI interview triggers a **deterministic, multi-stage reasoning pipeline** inside `features/ai/`. This pipeline is **orchestration code** — it lives in the feature layer, not in any engine, not in the AI provider. The AI model is called at specific stages; the rest is deterministic logic.
   352→
   353→### 3.1 Pipeline stages
   354→
   355→```
   356→User message received
   357→  │
   358→  ▼
   359→┌──────────────────────────────────────────────────────────────────┐
   360→│ STAGE 1 — Response Parsing                                       │
   361→│                                                                  │
   362→│ Parse the raw user message into a structured intermediate         │
   363→│ representation. This is NOT field extraction — it is parsing the  │
   364→│ user's intent and content into a form the next stages can        │
   365→│ process.                                                         │
   366→│                                                                  │
   367→│ Responsibilities:                                                │
   368→│ • Detect if the user is answering a question, asking a           │
   369→│   question, correcting a previous answer, or requesting          │
   370→│   navigation (e.g., "go back", "show me what you have").         │
   371→│ • Detect "I don't know" / uncertain responses.                   │
   372→│ • Detect corrections and contradictions with previous answers.    │
   373→│ • Detect when the user provides information for a field not      │
   374→│   currently being asked about (opportunistic extraction).        │
   375→│ Output: ParsedUserResponse { intent, content, targetField?,      │
   376→│         uncertainty, corrections? }                               │
   377→└───────────────────────────┬──────────────────────────────────────┘
   378→                            │
   379→                            ▼
   380→┌──────────────────────────────────────────────────────────────────┐
   381→│ STAGE 2 — Structured Field Extraction                            │
   382→│                                                                  │
   383→│ Extract structured field values from the parsed response.         │
   384→│ This is the stage where the AI model is called to map natural    │
   385→│ language to profile fields.                                      │
   386→│                                                                  │
   387→│ Responsibilities:                                                │
   388→│ • Map user's natural-language answer to one or more              │
   389→│   ProjectProfile fields.                                         │
   390→│ • Handle partial answers (e.g., "about 5 lakhs" → 500000 with    │
   391→│   low extractConfidence score).                                  │
   392→│ • Handle multi-field answers (e.g., "I'm 32, from Anantapur,    │
   393→│   SC category" → three fields extracted).                         │
   394→│ • When the user says "I don't know" or gives an uncertain        │
   395→│   answer, produce no extraction — flag the field for suggestion   │
   396→│   (Stage 6).                                                     │
   397→│ • All extractions carry an extractConfidence score (0–1).         │
   398→│                                                                  │
   399→│ AI is called here. The AI receives the conversation context and   │
   400→│ the current profile state, and returns structured field           │
   401→│ extractions with provenance info.                                │
   402→│                                                                  │
   403→│ Output: FieldExtraction[] { fieldPath, value, extractConfidence,  │
   404→│         source: "AI" }                                           │
   405→└───────────────────────────┬──────────────────────────────────────┘
   406→                            │
   407→                            ▼
   408→┌──────────────────────────────────────────────────────────────────┐
   409→│ STAGE 3 — Profile Update                                         │
   410→│                                                                  │
   411→│ Apply extracted fields to the in-memory Structured Project        │
   412→│ Profile.                                                         │
   413→│                                                                  │
   414→│ Responsibilities:                                                │
   415→│ • Apply each FieldExtraction to the profile.                     │
   416→│ • If a field already has a value and the user is correcting it,  │
   417→│   update the value and set source=USER, verification=UNVERIFIED.  │
   418→│ • If the AI extracted a value but extractConfidence is below      │
   419→│   threshold, do NOT apply to the profile — instead, present it   │
   420→│   as a suggestion for user confirmation (see Section 8).          │
   421→│ • Track provenance: every applied field gets a FieldProvenance   │
   422→│   entry with source and verification (see Section 9).         │
   423→│ • Trigger derived-field updates (e.g., if gender changes,         │
   424→│   re-derive isWomen).                                            │
   425→│ Output: Updated ProjectProfile                                   │
   426→└───────────────────────────┬──────────────────────────────────────┘
   427→                            │
   428→                            ▼
   429→┌──────────────────────────────────────────────────────────────────┐
   430→│ STAGE 4 — Validation                                             │
   431→│                                                                  │
   432→│ Run the Validation Engine against the updated profile.           │
   433→│                                                                  │
   434→│ Responsibilities:                                                │
   435→│ • Compute completeness percentage.                               │
   436→│ • Identify missing mandatory fields.                             │
   437→│ • Run Zod and business-rule validation on all populated fields.  │
   438→│ • Detect contradictions between fields.                          │
   439→│ • Update provenance metadata (see Section 9).                    │
   440→│                                                                  │
   441→│ This calls engines/validation-engine — a pure, deterministic,    │
   442→│ independently testable engine. The AI is NOT involved here.      │
   443→│                                                                  │
   444→│ Output: ValidationState { completeness, missingFields,           │
   445→│         errors, contradictions, updated provenance }              │
   446→└───────────────────────────┬──────────────────────────────────────┘
   447→                            │
   448→                            ▼
   449→┌──────────────────────────────────────────────────────────────────┐
   450→│ STAGE 5 — Dependency Evaluation                                 │
   451→│                                                                  │
   452→│ Evaluate the field dependency graph (see Section 7) to           │
   453→│ determine what downstream fields are now answerable or need      │
   454→│ revision.                                                        │
   455→│                                                                  │
   456→│ Responsibilities:                                                │
   457→│ • When a field changes, check which other fields depend on it    │
   458→│   (e.g., NIC code fixed → sector fixed → cost ceiling known).   │
   459→│ • If a dependency change invalidates a previously-set field,      │
   460→│   flag it for re-confirmation (but do NOT silently clear it).    │
   461→│ • Determine which interview phase we are now in based on         │
   462→│   completed dependencies.                                        │
   463→│ • When activity resolution completes (NIC code fixed),           │
   464→│   trigger Knowledge Engine lookups for machinery, raw materials,  │
   465→│   and process defaults.                                          │
   466→│ Output: DependencyEffects { invalidatedFields, newPhase?,        │
   467→│         knowledgeQueries? }                                       │
   468→└───────────────────────────┬──────────────────────────────────────┘
   469→                            │
   470→                            ▼
   471→┌──────────────────────────────────────────────────────────────────┐
   472→│ STAGE 6 — Knowledge Retrieval & Suggestion Generation            │
   473→│                                                                  │
   474→│ Query the Knowledge Package (via Knowledge Engine) for           │
   475→│ contextual data and suggestions.                                 │
   476→│                                                                  │
   477→│ Responsibilities:                                                │
   478→│ • If the user gave an uncertain/"I don't know" answer,          │
   479→│   query the Knowledge Package for suggested values               │
   480→│   (machinery lists, indicative costs, raw material defaults,     │
   481→│   wage benchmarks).                                              │
   482→│ • If a new NIC code was fixed, retrieve associated machinery,    │
   483→│   raw materials, process descriptions, and indicative costs.     │
   484→│ • If the user is in PROJECT_SIZING, retrieve benchmark           │
   485→│   project costs for similar activities.                           │
   486→│ • Suggestions carry their source and are NEVER treated as        │
   487→│   facts — they are presented to the user for confirmation.       │
   488→│                                                                  │
   489→│ This calls engines/knowledge-engine — a pure, deterministic       │
   490→│ engine that searches the Knowledge Package data.                 │
   491→│                                                                  │
   492→│ Output: Suggestion[] { fieldPath, suggestedValue, source,        │
   493→│         reasoning }                                               │
   494→└───────────────────────────┬──────────────────────────────────────┘
   495→                            │
   496→                            ▼
   497→┌──────────────────────────────────────────────────────────────────┐
   498→│ STAGE 7 — Question Planning                                      │
   499→│                                                                  │
   500→│ Determine the next question the AI should ask.                   │
   501→│                                                                  │
   502→│ Responsibilities:                                                │
   503→│ • Read Validation State: prioritize missing mandatory fields.     │
   504→│ • Read dependency graph: only ask questions whose dependencies   │
   505→│   are satisfied.                                                  │
   506→│ • Read interview phase: follow the phase sequence (Section 4).  │
   507→│ • If there are suggestions pending (from Stage 6), the next      │
   508→│   "question" may be a suggestion-present-and-confirm step.       │
   509→│ • If there are contradictions, prioritize resolving those        │
   510→│   before collecting new fields.                                   │
   511→│ • If validation errors exist on recently-changed fields,         │
   512→│   present them for correction before moving on.                   │
   513→│ • If completeness >= threshold and no blockers remain,            │
   514→│   transition to the REVIEW or VALIDATION_COMPLETION phase.       │
   515→│ • The planned question maps to a specific field or field group    │
   516→│   in the ProjectProfile schema (principle 15).                   │
   517→│                                                                  │
   518→│ Output: PlannedQuestion { targetField, questionType,              │
   519→│         suggestions?, context, phase }                            │
   520→│                                                                  │
   521→│ QuestionType = "FIELD_COLLECTION" | "SUGGESTION_CONFIRMATION"     │
   522→│              | "CONTRADICTION_RESOLUTION" | "ERROR_CORRECTION"    │
   523→│              | "PHASE_TRANSITION" | "REVIEW"                      │
   524→└───────────────────────────┬──────────────────────────────────────┘
   525→                            │
   526→                            ▼
   527→┌──────────────────────────────────────────────────────────────────┐
   528→│ STAGE 8 — Natural-Language Response Generation                   │
   529→│                                                                  │
   530→│ Convert the planned question (and any suggestions, validation    │
   531→│ feedback, or context) into a natural-language response to the    │
   532→│ user.                                                            │
   533→│                                                                  │
   534→│ Responsibilities:                                                │
   535→│ • Generate a conversational question that maps to the planned    │
   536→│   target field(s).                                               │
   537→│ • If suggestions exist, present them with clear labeling:         │
   538→│   "Based on similar [activity] units, typical machinery includes  │
   539→│   X, Y, Z. Would you like to use these or add your own?"         │
   540→│ • If there are validation errors, explain them in user-friendly   │
   541→│   language.                                                      │
   542→│ • If transitioning phases, provide a brief summary of what was    │
   543→│   collected and what comes next.                                  │
   544→│ • Maintain conversational warmth while staying focused on         │
   545→│   schema-driven data collection.                                  │
   546→│ • Never ask a question not mapped to the ProjectProfile schema.  │
   547→│                                                                  │
   548→│ AI is called here to generate the natural-language response.      │
   549→│ The planned question provides the structure; the AI provides      │
   550→│ the phrasing.                                                    │
   551→│                                                                  │
   552→│ Output: string (the message displayed to the user)                │
   553→└──────────────────────────────────────────────────────────────────┘
   554→```
   555→
   556→### 3.2 Where AI is and is not called
   557→
   558→| Stage | AI called? | Why |
   559→|-------|-----------|-----|
   560→| 1. Response Parsing | Optional | Can be rule-based for simple intents; AI for ambiguous cases |
   561→| 2. Field Extraction | Yes | Natural language → structured fields requires language understanding |
   562→| 3. Profile Update | No | Pure deterministic logic |
   563→| 4. Validation | No | Validation Engine — deterministic |
   564→| 5. Dependency Evaluation | No | Pure graph traversal |
   565→| 6. Knowledge Retrieval | No | Knowledge Engine — deterministic search |
   566→| 7. Question Planning | No | Pure logic reading validation state |
   567→| 8. Response Generation | Yes | Structured plan → natural language |
   568→
   569→AI is called at most twice per turn (stages 2 and 8). The other six stages are deterministic and testable without any AI model.
   570→
   571→### 3.3 Prompt construction
   572→
   573→The prompts sent to the AI provider include:
   574→
   575→- **The system prompt** (from the Knowledge Package's `default prompt templates`).
   576→- **The relevant portion of the ProjectProfile** — only fields relevant to the current phase and the planned question. PII is minimized; the full profile is never sent.
   577→- **The planned question metadata** (target field, question type, suggestions).
   578→- **Any recent conversation context** needed for coherence (last N turns, not the full history).
   579→
   580→PII redaction is applied before any data enters the prompt. See [13-security-and-privacy.md](13-security-and-privacy.md).

   584→## 4. Interview Phases
   585→
   586→The interview is not one long, unstructured conversation. It progresses through **named phases**, each with a clear objective. Phases are sequential — a phase does not begin until the previous phase's objective is substantially met. Within a phase, the AI may ask questions in any order that feels natural, but it cannot skip ahead to fields whose dependencies are unsatisfied.
   587→
   588→### Phase 1 — Applicant Discovery
   589→
   590→**Objective:** Establish who the applicant is and their eligibility category.
   591→
   592→Fields collected: `applicant.name`, `applicant.age`, `applicant.gender`, `applicant.category`, `applicant.entityType`, `applicant.education`, `applicant.experienceYears`, `applicant.priorSubsidy`, `applicant.edpCompleted`.
   593→
   594→Exit condition: Applicant identity and category are known. Eligibility pre-checks (age, prior subsidy) can be evaluated. If the applicant is clearly ineligible (e.g., age < 18), the interview surfaces this immediately — the user is informed but not blocked from continuing (they may wish to review or adjust).
   595→
   596→**Why this is first:** Category and area determine subsidy rates and cost ceilings — everything downstream depends on knowing who the applicant is.
   597→
   598→### Phase 2 — Business Discovery
   599→
   600→**Objective:** Understand what the applicant wants to do, in their own words, and where.
   601→
   602→Fields collected: `business.description`, `business.name`, `location.state`, `location.district`, `location.area` (urban/rural), `location.mandal`, `location.village`, `location.pinCode`.
   603→
   604→Exit condition: A natural-language business description exists and the geographic context is established. The AI has enough to begin activity resolution.
   605→
   606→**Why this is second:** The business description is the input to activity resolution (Phase 3). The location determines the area classification (urban/rural) which affects subsidy rates.
   607→
   608→### Phase 3 — Activity Resolution
   609→
   610→**Objective:** Map the business description to a specific NIC code, sector, and sub-category. This is the **critical juncture** where the Knowledge Engine is most heavily involved.
   611→
   612→Process: See Section 6 (Activity Discovery Lifecycle) for the full specification.
   613→
   614→Exit condition: `business.nicCode`, `business.nicDescription`, `business.sector`, and `business.subCategory` are fixed. PMEGP rules for this sector/category/area combination are now determined.
   615→
   616→**Why this is third:** The NIC code determines the sector, which determines the project cost ceiling, eligible subsidy rate, and what machinery/raw materials are relevant. No financial planning can proceed without this.
   617→
   618→### Phase 4 — Project Sizing
   619→
   620→**Objective:** Determine the physical scale of the project — land, building, machinery, employees, production capacity.
   621→
   622→Fields collected: `land.*`, `capacity.*`, `machinery.*` (items), `employees.*`, `utilities.*`, `attachments.*` (if quotation scanning).
   623→
   624→Exit condition: Machinery list is populated (with at least indicative items), employee count is known, production capacity is defined, and land/building status is clear.
   625→
   626→**Why this is fourth:** Machinery and building costs drive the capital expenditure, which is the largest component of project cost. The Knowledge Package assists here with activity-specific machinery and raw material suggestions.
   627→
   628→### Phase 5 — Financial Planning
   629→
   630→**Objective:** Determine the financial parameters — working capital, interest rate, loan tenure, projected sales.
   631→
   632→Fields collected: `rawMaterials.*`, `financials.*`, `workingCapitalDetail.*`, `market.*`.
   633→
   634→Exit condition: Total project cost can be computed. Working capital is determined. Interest rate and tenure are set. Projected monthly sales are estimated.
   635→
   636→**Why this is fifth:** Financial planning requires knowing the capital expenditure (from Phase 4) to compute total project cost, working capital, and the subsidy/loan/own-contribution split.
   637→
   638→### Phase 6 — Review
   639→
   640→**Objective:** Present the complete profile to the user for review and correction. This is a human-in-the-loop checkpoint before any engine runs.
   641→
   642→Behavior:
   643→- Display the structured profile in a readable, sectioned format.
   644→- Highlight any fields sourced from AI, OCR, or Knowledge Package (these benefit most from review, though all fields are confirmed together on the REVIEW_PENDING screen).
   645→- Highlight any validation warnings or soft contradictions.
   646→- Allow the user to edit any field.
   647→- Show a completeness summary and list of any remaining issues.
   648→
   649→Exit condition: The user explicitly confirms the profile is correct, or edits are applied and validation is re-run.
   650→
   651→**Why this exists:** The beneficiary must see and approve the data before engines produce numbers that go into a bank document. This is the last point where data quality is a human responsibility.
   652→
   653→### Phase 7 — Validation Completion
   654→
   655→**Objective:** Run the final Validation Engine check. If the profile passes (completeness at threshold, no blocking errors, no unresolved contradictions), unlock downstream engines.
   656→
   657→Behavior:
   658→- The Validation Engine runs one final time.
   659→- If blockers exist, return to the relevant phase to resolve them.
   660→- If the profile passes, transition the project status to "validated" and unlock Eligibility → Financial → DPR engines.
   661→- The user is informed that the project is ready for processing.
   662→
   663→Exit condition: Validation Engine passes. Project status becomes "validated." Downstream engines are unlocked.
   664→
   665→**Why this is last:** This is the gate described in Design Principle 14 and doc 04 §5. No engine runs until this passes.
   666→

   669→## 5. Knowledge-Assisted Interview
   670→
   671→Most beneficiaries do not know their NIC code, what machinery they need, or what their working capital should be. They know: "I want to start a poultry farm" or "I want to buy a goods vehicle." The Knowledge Package bridges this gap.
   672→
   673→### 5.1 How it works
   674→
   675→When the beneficiary provides a business description or cannot answer a technical question, the interview orchestration queries the Knowledge Package through the Knowledge Engine:
   676→
   677→```
   678→Beneficiary says: "I want to do poultry farming."
   679→
   680→    ↓ (features/ai/ calls Knowledge Engine)
   681→
   682→Knowledge Engine searches:
   683→  • activities/NIC codes → "Poultry farming" matches NIC 0146xx
   684→  • machinery database → "Poultry sheds, feeders, waterers, incubator..."
   685→  • raw material database → "Feed, chicks, vaccines, medicines..."
   686→  • financial benchmarks → "Typical project cost for 1000-bird unit: ₹X lakhs"
   687→  • FAQ → "Common questions about poultry farming under PMEGP"
   688→
   689→    ↓ (Knowledge Engine returns structured results)
   690→
   691→Interview presents:
   692→  "Poultry farming falls under NIC 014612 (Poultry farming). This is classified
   693→   as a Service sector activity with a maximum project cost of ₹25 lakhs.
   694→
   695→   Based on similar projects, typical machinery includes:
   696→   1. Poultry shed (2000 sq ft) — ₹3,50,000
   697→   2. Feeders (20 units) — ₹20,000
   698→   3. Waterers (20 units) — ₹15,000
   699→   4. Incubator (1 unit) — ₹50,000
   700→
   701→   Would you like to use these as a starting point?"
   702→
   703→    ↓ (User confirms or edits)
   704→
   705→Only confirmed values enter the Structured Project Profile.
   706→Suggestions are never treated as facts.
   707→```
   708→
   709→### 5.2 Rules for knowledge assistance
   710→
   711→1. **Knowledge queries are initiated by the interview, not by the user.** The user does not interact with the Knowledge Engine directly. The AI interview decides when to query and what to query for.
   712→
   713→2. **Suggestions are presented, not applied.** Every suggestion from the Knowledge Package is shown to the user with a clear label ("Based on typical [activity] projects...") and requires explicit acceptance. See Section 10 (Suggestion Lifecycle) and Section 9 (Provenance Model).
   714→
   715→3. **The user can always override.** A suggested machinery list, raw material cost, or working capital amount is a starting point — the user can add, remove, or edit any item.
   716→
   717→4. **Knowledge assists; it does not decide.** The Knowledge Package does not determine eligibility, calculate subsidy, or compute financials. It provides reference data that the interview presents for human confirmation.
   718→
   719→5. **Knowledge queries are deterministic and offline.** The Knowledge Engine searches bundled data; no network call is required. If AI is unavailable, the guided forms present the same Knowledge Package data as pre-populated defaults in the form fields.

   723→## 6. Activity Discovery Lifecycle
   724→
   725→Activity discovery is the process of mapping the beneficiary's business description to a specific NIC code, sector, and sub-category. This is a **responsibility of the Knowledge Engine** (as specified in [AGENTS.md](../AGENTS.md)) — no new engine is introduced.
   726→
   727→### 6.1 The lifecycle
   728→
   729→```
   730→┌──────────────────────────────────────────────────────────────────┐
   731→│ STEP 1 — Business Description Received                           │
   732→│                                                                  │
   733→│ The beneficiary provides a natural-language description of       │
   734→│ what they want to do. This happens during Phase 2 (Business     │
   735→│ Discovery) and flows into Phase 3.                              │
   736→│                                                                  │
   737→│ Example: "I want to start a small bakery in my village."        │
   738→└───────────────────────────┬──────────────────────────────────────┘
   739→                            │
   740→                            ▼
   741→┌──────────────────────────────────────────────────────────────────┐
   742→│ STEP 2 — Knowledge Engine Search                                 │
   743→│                                                                  │
   744→│ The interview orchestration sends the business description to    │
   745→│ the Knowledge Engine, which searches the NIC code database       │
   746→│ using:                                                          │
   747→│ • Exact text matching against NIC descriptions                  │
   748→│ • Text search (fuzzy matching, keyword extraction)               │
   749→│ • Synonym expansion (e.g., "bakery" → "bread making",           │
   750→│   "confectionery production")                                   │
   751→│                                                                  │
   752→│ The Knowledge Engine returns a ranked list of candidate NIC      │
   753→│ activities.                                                     │
   754→└───────────────────────────┬──────────────────────────────────────┘
   755→                            │
   756→                            ▼
   757→┌──────────────────────────────────────────────────────────────────┐
   758→│ STEP 3 — Candidate NIC Activities Presented                      │
   759→│                                                                  │
   760→│ The interview presents the top candidates to the beneficiary:    │
   761→│                                                                  │
   762→│ "We found these matching activities:                            │
   763→│  1. [10701] Manufacture of bread, cakes, pastries (Manufacturing)│
   764→│  2. [10702] Manufacture of biscuits and cookies (Manufacturing) │
   765→│  3. [56211] Retail sale of bread and bakery products (Service — │
   766→│     Trading)                                                     │
   767→│                                                                  │
   768→│  Are you planning to manufacture bakery products, or sell them?" │
   769→└───────────────────────────┬──────────────────────────────────────┘
   770→                            │
   771→                            ▼
   772→┌──────────────────────────────────────────────────────────────────┐
   773→│ STEP 4 — User Confirmation                                       │
   774→│                                                                  │
   775→│ The user selects (or clarifies) which activity matches their     │
   776→│ intention. The user may also describe further if none of the     │
   777→│ candidates match.                                                │
   778→│                                                                  │
   779→│ If no match is found, the interview asks follow-up questions     │
   780→│ to narrow down the search (e.g., "Are you making products or     │
   781→│ providing a service?").                                          │
   782→└───────────────────────────┬──────────────────────────────────────┘
   783→                            │
   784→                            ▼
   785→┌──────────────────────────────────────────────────────────────────┐
   786→│ STEP 5 — NIC Code Fixed                                          │
   787→│                                                                  │
   788→│ Once the user confirms a NIC code:                              │
   789→│ • business.nicCode is set                                        │
   790→│ • business.nicDescription is set                                 │
   791→│ • business.sector is derived from the NIC entry                  │
   792→│ • business.subCategory is derived from the NIC entry             │
   793→│ • business.activityType is derived (MANUFACTURING or SERVICE)    │
   794→│                                                                  │
   795→│ These fields are set with source=USER, verification=CONFIRMED      │
   796→│ in the provenance metadata. They become authoritative.             │
   797→└───────────────────────────┬──────────────────────────────────────┘
   798→                            │
   799→                            ▼
   800→┌──────────────────────────────────────────────────────────────────┐
   801→│ STEP 6 — PMEGP Rules Determined                                  │
   802→│                                                                  │
   803→│ With sector and subCategory fixed, the applicable PMEGP rules    │
   804→│ are now known:                                                   │
   805→│ • Project cost ceiling (Manufacturing: ₹50 lakhs,                 │
   806→│   Service: ₹25 lakhs)                                           │
   807→│ • Subsidy rate (from category × area matrix)                     │
   808→│ • Own-contribution percentage                                   │
   809→│ • Eligibility criteria for this activity type                    │
   810→│ • Negative-list check (if applicable)                            │
   811→│                                                                  │
   812→│ These rules are NOT stored in the profile — they are looked up   │
   813→│ from the Knowledge Package by the engines when they run. The     │
   814→│ profile stores only the beneficiary's data and their chosen       │
   815→│ activity.                                                        │
   816→└───────────────────────────┬──────────────────────────────────────┘
   817→                            │
   818→                            ▼
   819→┌──────────────────────────────────────────────────────────────────┐
   820→│ STEP 7 — Interview Continues                                      │
   821→│                                                                  │
   822→│ With the NIC code fixed, the interview transitions to Phase 4    │
   823→│ (Project Sizing). The Knowledge Engine now has the context to    │
   824→│ provide activity-specific suggestions for machinery, raw         │
   825→│ materials, and benchmarks.                                       │
   826→└──────────────────────────────────────────────────────────────────┘
   827→```
   828→
   829→### 6.2 Ambiguity and multi-match handling
   830→
   831→- If multiple NIC codes match with similar relevance, the interview presents all candidates and asks the user to choose.
   832→- If no NIC code matches, the interview asks clarifying questions (manufacturing vs service, product vs service delivery, etc.) and re-searches.
   833→- The interview never silently assigns a NIC code. The user must confirm.
   834→- If the beneficiary's description is too vague for any match, the interview explains what information is needed and asks follow-up questions.
   835→
   836→### 6.3 Trading and Transport
   837→
   838→Trading and Transport are not standalone sectors in PMEGP — they are sub-categories under Service. The activity discovery lifecycle handles this transparently:
   839→
   840→- A beneficiary saying "I want to start a general store" is matched to a Trading NIC code under the Service sector.
   841→- A beneficiary saying "I want to buy a goods vehicle" is matched to a Transport NIC code under the Service sector.
   842→- The sector and cost ceiling are correctly determined (Service: ₹25 lakhs) even though the activity is Trading or Transport.
   843→- The user never needs to know this distinction — the Knowledge Engine resolves it.
   844→

   847→## 7. Field Dependency Graph
   848→
   849→The interview cannot ask questions in arbitrary order. Many fields depend on other fields being known first. The dependency graph enforces this ordering.
   850→
   851→### 7.1 Core dependency chains
   852→
   853→```
   854→applicant.category + location.area
   855→    → subsidy rate (determined by Eligibility/Financial Engine, not stored)
   856→
   857→business.nicCode
   858→    → business.sector
   859→    → business.subCategory
   860→    → business.activityType
   861→    → project cost ceiling (from Knowledge Package, used by engines)
   862→    → eligible machinery (from Knowledge Package)
   863→    → eligible raw materials (from Knowledge Package)
   864→
   865→location.area
   866→    → subsidy rate determination
   867→
   868→machinery.items[]
   869→    → financials.machineryAndEquipment
   870→    → financials.totalFixedCapital
   871→    → financials.totalProjectCost
   872→
   873→land.buildingType + land.constructionCost
   874→    → financials.buildingAndCivilWorks
   875→    → financials.totalFixedCapital
   876→    → financials.totalProjectCost
   877→
   878→rawMaterials.items[]
   879→    → rawMaterials.totalMonthlyCost
   880→    → working capital computation
   881→    → financials.workingCapital
   882→    → financials.totalProjectCost
   883→
   884→financials.totalProjectCost
   885→    → financials.totalFixedCapital + financials.workingCapital
   886→    → Eligibility Engine (cost.ceiling check)
   887→    → Financial Engine (subsidy, loan, EMI, DSCR, P&L)
   888→
   889→capacity.installedCapacity + capacity.projectedCapacityUtil
   890→    → projected monthly production
   891→    → market.sellingPricePerUnit → projectedMonthlySales
   892→    → Financial Engine (revenue projections)
   893→
   894→employees.*
   895→    → employees.totalMonthlyWages
   896→    → utilities + wages = operating expenses
   897→    → Financial Engine (P&L, cash flow)
   898→```
   899→
   900→### 7.2 Dependency rules for the interview
   901→
   902→1. **A field is only asked when all its dependencies are satisfied.** The question planner (Stage 7 of the pipeline) checks the dependency graph before selecting the next question.
   903→
   904→2. **If a dependency changes, dependents are flagged.** For example, if the user changes the NIC code (going back to Phase 3), all machinery, raw material, and financial data that was based on the old NIC code is flagged for review — not silently cleared.
   905→
   906→3. **The dependency graph is a read-only reference structure.** It is defined once (in the Knowledge Package or in `shared/types`) and consumed by the interview orchestration and the Validation Engine. It is not modified at runtime.
   907→
   908→4. **Dependencies flow forward only.** A field in a later phase never determines a field in an earlier phase. This ensures the interview never needs to "go back" for data — though the user can always manually navigate back to edit.
   909→
   910→5. **The guided wizard follows the same dependency order.** The wizard's step sequence mirrors the phase structure, and the Validation Engine's `missingFields` output respects the same dependency graph.

   914→## 8. "I Don't Know" Strategy
   915→
   916→Most PMEGP beneficiaries are first-time entrepreneurs from rural areas. They often cannot answer questions about machinery costs, working capital requirements, or financial projections. The application must handle this gracefully without blocking progress or degrading the quality of the final DPR.
   917→
   918→### 8.1 The flow
   919→
   920→```
   921→User says "I don't know" / "Mujhe nahi pata" / gives an uncertain answer
   922→  │
   923→  ▼
   924→┌──────────────────────────────────────────────────────────────────┐
   925→│ STEP 1 — Detect Uncertainty                                      │
   926→│                                                                  │
   927→│ Stage 1 (Response Parsing) detects the uncertainty. The field   │
   928→│ extraction (Stage 2) produces NO extraction for this field.      │
   929→│ The field remains in its current state (MISSING or unchanged).  │
   930→└───────────────────────────┬──────────────────────────────────────┘
   931→                            │
   932→                            ▼
   933→┌──────────────────────────────────────────────────────────────────┐
   934→│ STEP 2 — Knowledge Package Suggestion                           │
   935→│                                                                  │
   936→│ Stage 6 (Knowledge Retrieval) queries the Knowledge Package      │
   937→│ for a suggested value based on:                                 │
   938→│ • The current NIC code / activity type                          │
   939→│ • The beneficiary's location (district/state benchmarks)        │
   940→│ • Activity-specific defaults from the Knowledge Package          │
   941→│ • Financial benchmarks for similar project sizes                │
   942→│                                                                  │
   943→│ The Knowledge Engine returns a structured suggestion with       │
   944→│ source attribution.                                             │
   945→└───────────────────────────┬──────────────────────────────────────┘
   946→                            │
   947→                            ▼
   948→┌──────────────────────────────────────────────────────────────────┐
   949→│ STEP 3 — AI Explains the Suggestion                             │
   950→│                                                                  │
   951→│ Stage 8 (Response Generation) produces a natural-language        │
   952→│ explanation of the suggestion:                                  │
   953→│                                                                  │
   954→│ "That's completely fine! For a bakery unit in a rural area,     │
   955→│  typical machinery costs around ₹2-3 lakhs. Based on common      │
   956→│  setups, here's what most people include:                       │
   957→│  • Oven — ₹80,000                                                │
   958→│  • Dough mixer — ₹45,000                                         │
   959→│  • Display counter — ₹25,000                                     │
   960→│                                                                  │
   961→│  Would you like to start with these and adjust later?"           │
   962→└───────────────────────────┬──────────────────────────────────────┘
   963→                            │
   964→                            ▼
   965→┌──────────────────────────────────────────────────────────────────┐
   966→│ STEP 4 — User Reviews and Decides                                │
   967→│                                                                  │
   968→│ The user has three options:                                      │
   969→│                                                                  │
   970→│ a) ACCEPT — The suggested value(s) are applied to the profile     │
   971→│    with source=KNOWLEDGE, verification=UNVERIFIED.                │
   972→│                                                                  │
   973→│ b) EDIT — The user provides their own value, which is applied    │
   974→│    with source=USER, verification=UNVERIFIED.                     │
   975→│                                                                  │
   976→│ c) SKIP — The field remains source=null, verification=UNVERIFIED  │
   977→│    (equivalent to MISSING). The interview moves on.               │
   978→│    The Validation Engine will flag this field as missing. The    │
   979→│    profile cannot pass validation until all mandatory fields are  │
   980→│    engine-ready (see Section 9).
   981→└───────────────────────────┬──────────────────────────────────────┘
   982→                            │
   983→                            ▼
   984→┌──────────────────────────────────────────────────────────────────┐
   985→│ STEP 5 — Provenance Tracking                                       │
   986→│                                                                  │
   987→│ Regardless of the user's choice, the provenance metadata is      │
   988→│ updated:                                                         │
   989→│                                                                  │
   990→│ • Accepted suggestion → source=KNOWLEDGE, verification=UNVERIFIED  │
   991→│ • User-provided value → source=USER, verification=UNVERIFIED      │
   992→│ • Skipped → source=null, verification=UNVERIFIED                  │
   993→│                                                                  │
   994→│ The Validation Engine reads provenance metadata to compute       │
   995→│ aggregate provenance score (see Section 9). Only engine-ready     │
   996→│ fields (source=USER, or verification=CONFIRMED) contribute fully. │
   997→│ Non-engine-ready fields with a value contribute at reduced weight. │
   998→│ MISSING fields contribute zero.                                   │
   999→└──────────────────────────────────────────────────────────────────┘
  1000→```
  1001→
  1002→### 8.2 Rules for unknown answers
  1003→
  1004→1. **"I don't know" is never an error.** The interview treats it as a normal response and offers assistance.
  1005→
  1006→2. **The user is never forced to provide a value immediately.** They can skip and come back. The Validation Engine will eventually require all mandatory fields, but the interview does not block mid-conversation.
  1007→
  1008→3. **Suggestions are based on the current activity context.** A bakery suggestion is different from a poultry suggestion. The Knowledge Engine uses the fixed NIC code (if available) or the business description to provide relevant defaults.
  1009→
  1010→4. **Suggested values are clearly labeled.** The user must always know which values they entered and which were suggested. The Review phase (Phase 6) highlights all suggested values for explicit confirmation.
  1011→
  1012→5. **The interview does not invent financial figures.** Even when suggesting machinery costs or working capital, the values come from the Knowledge Package — never from the AI model's parametric knowledge. The AI's role is to phrase the suggestion naturally, not to generate the number.
  1013→

  1016→## 9. Provenance Model (Source × Verification)
  1017→
  1018→Every field in the Structured Project Profile carries **provenance metadata** — two orthogonal dimensions that track where a value came from and whether it has been confirmed. This separation provides much richer provenance than a single status field.
  1019→
  1020→### 9.1 Why two dimensions?
  1021→
  1022→The old single-status model (`USER_CONFIRMED`, `KNOWLEDGE_SUGGESTED`, etc.) conflated two distinct concepts:
  1023→
  1024→- **Provenance** (where did this value originate?) — important for audit trails, DPR annotation, and understanding data quality.
  1025→- **Verification** (has the beneficiary explicitly agreed this is correct?) — important for the validation gate and engine readiness.
  1026→
  1027→By separating them, we can express any combination:
  1028→
  1029→| Example field | Source | Verification | Meaning |
  1030→|---|---|---|---|
  1031→| Applicant name (typed) | `USER` | `UNVERIFIED` | User typed it directly — has a value but not yet confirmed via REVIEW_PENDING |
  1032→| Machinery cost (suggested) | `KNOWLEDGE` | `UNVERIFIED` | Knowledge Package suggested it, user accepted but hasn't confirmed via REVIEW_PENDING |
  1033→| Machinery cost (confirmed) | `KNOWLEDGE` | `CONFIRMED` | Knowledge Package suggested it, user confirmed inline or via REVIEW_PENDING |
  1034→| Aadhaar number (OCR) | `OCR` | `CONFIRMED` | Extracted from document scan, user confirmed inline or via REVIEW_PENDING |
  1035→| Business description (AI) | `AI` | `CONFIRMED` | AI extracted from conversation, user confirmed via REVIEW_PENDING |
  1036→| Any field (post-gate) | *any* | `VALIDATED` | Validation Engine has stamped this field after the profile passed validation |
  1037→
  1038→### 9.2 Source dimension
  1039→
  1040→| Source | Meaning | Set by |
  1041→|---|---|---|
  1042→| `null` | No value exists — the field is MISSING | Initial state |
  1043→| `USER` | User directly typed, selected, or provided the value | Form input, guided wizard, or user editing a suggestion |
  1044→| `AI` | AI model extracted the value from a conversational response | Stage 2 of the reasoning pipeline |
  1045→| `OCR` | OCR Engine extracted the value from a scanned document | OCR Engine |
  1046→| `KNOWLEDGE` | Knowledge Package suggested this value; user accepted the suggestion | Knowledge Engine → user acceptance |
  1047→
  1048→### 9.3 Verification dimension
  1049→
  1050→| Verification | Meaning | Set by |
  1051→|---|---|---|
  1052→| `UNVERIFIED` | Value exists but has not been explicitly confirmed by the user | Default when a value is first written |
  1053→| `CONFIRMED` | User has explicitly reviewed and confirmed this value is correct | User action (in Review phase, or inline confirmation) |
  1054→| `VALIDATED` | The Validation Engine has stamped this field after the profile passed validation | Validation Engine (set on all fields when profile transitions to `VALIDATED`) |
  1055→
  1056→### 9.4 Engine-ready definition
  1057→
  1058→A field is **engine-ready** if and only if:
  1059→
  1060→```
  1061→engineReady(field) = (field.provenance.verification === "CONFIRMED")
  1062→                  || (field.provenance.verification === "VALIDATED")
  1063→```
  1064→
  1065→Source does not factor into engine readiness. A user-typed value and a Knowledge-suggested value are treated identically by the gate — both must be confirmed before engines can consume them.
  1066→
  1067→This is a deliberate design choice for a financial application:
  1068→
  1069→- **Typos are real.** A user can accidentally type ₹20,00,000 instead of ₹2,00,000. The REVIEW_PENDING screen catches this.
  1070→- **Source answers where it came from. Verification answers whether the system trusts it for downstream processing.** These are independent concerns — conflating them would mean trusting USER source without verification, which is inappropriate for subsidy calculations.
  1071→- **`CONFIRMED` is set by user action** — either inline (during the Review phase) or in bulk (by pressing "Confirm Project" on the REVIEW_PENDING screen).
  1072→- **`VALIDATED` is a system-level stamp.** Set by the Validation Engine on all fields when the profile passes the final gate. It means "this value passed Zod + business rules."
  1073→
  1074→### 9.5 Provenance transitions
  1075→
  1076→```
  1077→// Initial state (no value)
  1078→source=null, verification=UNVERIFIED  ─── MISSING
  1079→
  1080→// User provides value directly
  1081→──[user types or selects]──→ source=USER, verification=UNVERIFIED
  1082→
  1083→// AI extraction (confidence threshold determines whether to apply or suggest)
  1084→──[AI extracts, extractConfidence ≥ threshold]──→ source=AI, verification=UNVERIFIED  (applied to profile)
  1085→──[AI extracts, extractConfidence < threshold]──→ NOT applied; presented as suggestion
  1086→   ──[user confirms inline]──→ verification=CONFIRMED
  1087→   ──[user edits]──→ source=USER, verification=UNVERIFIED
  1088→   ──[user rejects]──→ source=null, verification=UNVERIFIED  (MISSING again)
  1089→
  1090→// OCR extraction
  1091→──[OCR extracts, user accepts]──→ source=OCR, verification=UNVERIFIED
  1092→   ──[user confirms inline]──→ verification=CONFIRMED
  1093→   ──[user edits]──→ source=USER, verification=UNVERIFIED
  1094→   ──[user rejects]──→ source=null, verification=UNVERIFIED  (MISSING)
  1095→
  1096→// Knowledge suggestion
  1097→──[user accepts suggestion]──→ source=KNOWLEDGE, verification=UNVERIFIED
  1098→   ──[user confirms inline]──→ verification=CONFIRMED
  1099→   ──[user edits]──→ source=USER, verification=UNVERIFIED
  1100→   ──[user rejects]──→ source=null, verification=UNVERIFIED  (MISSING)
  1101→
  1102→// User edits any existing value
  1103→──[user edits]──→ source=USER, verification=UNVERIFIED  (always resets to USER source)
  1104→
  1105→// User presses "Confirm Project" on REVIEW_PENDING screen
  1106→──[REVIEW_PENDING → VALIDATED transition]──→ verification=CONFIRMED  (ALL fields with values)
  1107→
  1108→// Validation Engine stamps all fields after final gate
  1109→──[profile reaches VALIDATED]──→ verification=VALIDATED  (all fields)
  1110→```
  1111→
  1112→Note: "Inline confirmation" during the Review phase (Phase 6) is optional — the user can confirm individual fields early if they wish. The mandatory confirmation point is the REVIEW_PENDING screen.
  1113→
  1114→### 9.6 Aggregate provenance score
  1115→
  1116→The Validation Engine computes an aggregate provenance score displayed to the user:
  1117→
  1118→```
  1119→aggregateProvenance = Σ(provenanceWeight(field) for each mandatory field)
  1120→                    / count(mandatory fields)
  1121→
  1122→where:
  1123→  provenanceWeight(source=null)                                = 0     // MISSING
  1124→  provenanceWeight(any, verification=VALIDATED)                 = 1.0   // post-gate
  1125→  provenanceWeight(any, verification=CONFIRMED)                = 1.0   // engine-ready
  1126→  provenanceWeight(any source≠null, verification=UNVERIFIED)   = 0.5   // has value, not confirmed
  1127→```
  1128→
  1129→Note: source does not appear in the weight formula. A `USER`/`UNVERIFIED` field and a `KNOWLEDGE`/`UNVERIFIED` field contribute identically. The REVIEW_PENDING screen is what upgrades all UNVERIFIED fields to CONFIRMED.
  1130→
  1131→A profile with 100% completeness but 50% aggregate provenance means all fields have values but none are confirmed — the REVIEW_PENDING screen will draw attention to this.
  1132→
  1133→### 9.7 Provenance and downstream engines
  1134→
  1135→Downstream engines (Eligibility, Financial, DPR) **do not consume provenance metadata**. They consume the field values. Provenance is an interview-layer concern — it determines *whether* a field is ready for engine consumption, not *how* the engine processes it.
  1136→
  1137→**However, the Validation Engine enforces a provenance gate.** A profile cannot transition from `REVIEW_PENDING` to `VALIDATED` (and thus unlock downstream engines) until all mandatory fields have verification=CONFIRMED. In practice, pressing "Confirm Project" on the REVIEW_PENDING screen stamps all fields CONFIRMED, so this gate passes unless the user edited a field back to UNVERIFIED. See Section 11 (state machine) for the full gate logic.
  1138→
  1139→This means:
  1140→- An `UNVERIFIED` value may exist in the profile during the interview (so the user can see and edit it), but it is **not** engine-ready.
  1141→- The Review phase (Phase 6) offers early inline confirmation for individual fields, but the mandatory checkpoint is the REVIEW_PENDING screen.
  1142→- The interview can progress past an unconfirmed field (to avoid blocking the conversation), but the profile cannot reach `VALIDATED` until the user presses "Confirm Project."
  1143→- When the Validation Engine runs, it returns `nonEngineReadyMandatoryFields` alongside `missingFields`. Both must be empty for the final gate to pass.
  1144→
  1145→This is the enforcement of the principle that **suggested values never reach deterministic engines until the user confirms them.**
  1146→
  1147→### 9.8 Provenance for DPR annotation
  1148→
  1149→The DPR Engine can use the provenance metadata to annotate the generated document:
  1150→
  1151→- Fields with `source=KNOWLEDGE` can be footnoted: *"Based on typical values for this activity."*
  1152→- Fields with `source=OCR` can be footnoted: *"Extracted from uploaded document."*
  1153→- Fields with `source=USER` need no annotation — they are the beneficiary's own values.
  1154→- The `knowledgeSource` field in `FieldProvenance` provides the specific Knowledge Package entry reference for traceability.
  1155→

  1158→## 10. Suggestion Lifecycle
  1159→
  1160→Suggestions are how the Knowledge Package assists beneficiaries who cannot answer technical questions on their own. The lifecycle ensures suggestions are helpful but never authoritative.
  1161→
  1162→```
  1163→┌──────────────────────────────────────────────────────────────────┐
  1164→│ 1. KNOWLEDGE PACKAGE                                              │
  1165→│                                                                  │
  1166→│    Contains reference data: machinery lists, raw material lists, │
  1167→│    indicative costs, wage benchmarks, financial ratios, etc.     │
  1168→│    This data is versioned, sourced, and read-only at runtime.    │
  1169→└───────────────────────────┬──────────────────────────────────────┘
  1170→                            │
  1171→                            ▼
  1172→┌──────────────────────────────────────────────────────────────────┐
  1173→│ 2. KNOWLEDGE ENGINE (engines/knowledge-engine/)                   │
  1174→│                                                                  │
  1175→│    Receives a query from the interview (e.g., "machinery for     │
  1176→│    NIC 10701") and searches the Knowledge Package.               │
  1177→│    Returns structured results — never free text.                  │
  1178→│    This is a deterministic, pure, offline search engine.         │
  1179→│                                                                  │
  1180→│    Output: SuggestionResult { items: KnowledgeItem[], source,    │
  1181→│            confidence }                                           │
  1182→└───────────────────────────┬──────────────────────────────────────┘
  1183→                            │
  1184→                            ▼
  1185→┌──────────────────────────────────────────────────────────────────┐
  1186→│ 3. SUGGESTION                                                     │
  1187→│                                                                  │
  1188→│    The interview orchestration wraps the Knowledge Engine        │
  1189→│    result into a Suggestion object that includes:                │
  1190→│    • The suggested value(s)                                       │
  1191→│    • The source (which Knowledge Package entry)                   │
  1192→│    • A reasoning string (for the AI to explain to the user)      │
  1193→│    • The target field path(s) in the ProjectProfile              │
  1194→│                                                                  │
  1195→│    The suggestion is NOT yet applied to the profile.             │
  1196→└───────────────────────────┬──────────────────────────────────────┘
  1197→                            │
  1198→                            ▼
  1199→┌──────────────────────────────────────────────────────────────────┐
  1200→│ 4. USER REVIEW                                                    │
  1201→│                                                                  │
  1202→│    The suggestion is presented to the user — either by the AI    │
  1203→│    interview (natural language) or by the guided forms (pre-     │
  1204→│    populated form fields with a "Suggested" badge).             │
  1205→│                                                                  │
  1206→│    The user sees:                                                │
  1207→│    • What value is being suggested                               │
  1208→│    • Why (e.g., "Based on typical bakery projects")              │
  1209→│    • Options: Accept, Edit, or Skip                              │
  1210→└───────────────────────────┬──────────────────────────────────────┘
  1211→                            │
  1212→                            ▼
  1213→┌──────────────────────────────────────────────────────────────────┐
  1214→│ 5. ACCEPTANCE                                                     │
  1215→│                                                                  │
  1216→│    a) ACCEPT: Value is written to the profile with               │
  1217→│       provenance.source = KNOWLEDGE, provenance.verification = UNVERIFIED. │
  1218→│                                                                  │
  1219→│    b) EDIT: User provides their own value. Written to profile    │
  1220→│       with provenance.source = USER, provenance.verification = UNVERIFIED. │
  1221→│                                                                  │
  1222→│    c) SKIP: Field remains source=null (MISSING). Interview continues. │
  1223→└───────────────────────────┬──────────────────────────────────────┘
  1224→                            │
  1225→                            ▼
  1226→┌──────────────────────────────────────────────────────────────────┐
  1227→│ 6. STRUCTURED PROJECT PROFILE                                     │
  1228→│                                                                  │
  1229→│    The profile now contains the value (if accepted or edited).   │
  1230→│    The provenance metadata records how it arrived.                │
  1231→└───────────────────────────┬──────────────────────────────────────┘
  1232→                            │
  1233→                            ▼
  1234→┌──────────────────────────────────────────────────────────────────┐
  1235→│ 7. VALIDATION ENGINE                                              │
  1236→│                                                                  │
  1237→│    Runs on the updated profile. Checks completeness,             │
  1238→│    validation rules, contradictions, and provenance.              │
  1239→│    Determines if the profile is ready for downstream engines.    │
  1240→└───────────────────────────┬──────────────────────────────────────┘
  1241→                            │
  1242→                            ▼
  1243→┌──────────────────────────────────────────────────────────────────┐
  1244→│ 8. DETERMINISTIC ENGINES                                         │
  1245→│                                                                  │
  1246→│    Eligibility Engine, Financial Engine, DPR Engine, PDF Engine.  │
  1247→│    These engines receive the profile values. They do NOT know    │
  1248→│    and do NOT care whether a value was user-entered or           │
  1249→│    knowledge-suggested. The provenance model is invisible to     │
  1250→│    engines.                                                       │
  1251→└──────────────────────────────────────────────────────────────────┘
  1252→```
  1253→
  1254→### 10.1 Inviolable rules
  1255→
  1256→1. **Suggestions must never bypass confirmation.** A suggestion that is not explicitly accepted by the user is never written to the profile. The AI interview cannot silently apply a suggestion.
  1257→
  1258→2. **The Knowledge Engine returns data, not decisions.** It returns matching items from the Knowledge Package. The interview (feature layer) decides what to present and how.
  1259→
  1260→3. **AI-generated suggestions are prohibited.** The AI model must not invent machinery lists, cost estimates, or financial benchmarks. All suggestions originate from the Knowledge Package. The AI's role is to present them in natural language, not to generate them.
  1261→
  1262→4. **Suggested values are provisional, not authoritative.** A `KNOWLEDGE`/`UNVERIFIED` value is written to the profile so the user can see and edit it, but it is **not** engine-ready. No field is engine-ready until verification=CONFIRMED (via the REVIEW_PENDING "Confirm Project" action; see Section 9.4). See Section 11 (state machine — the REVIEW_PENDING screen is the single confirmation point for all fields).
  1263→
  1264→5. **The Review phase is the early safety net.** Phase 6 (Review) presents AI/OCR/KNOWLEDGE-sourced fields prominently so the user can catch issues early. But the mandatory confirmation point for ALL fields is the REVIEW_PENDING screen.
  1265→

  1268→## 11. Project Profile State Machine
  1269→
  1270→The interview phases (Section 4) describe *what* the interview does at each stage. The state machine describes *how the profile evolves* as a whole — including resume, edit, validation, and engine execution. This is the single source of truth for what transitions are legal and what each state means for the rest of the application.
  1271→
  1272→### 11.1 States
  1273→
  1274→```
  1275→EMPTY
  1276→  │
  1277→  ▼
  1278→PARTIAL
  1279→  │
  1280→  ▼
  1281→DISCOVERING
  1282→  │
  1283→  ▼
  1284→COMPLETE
  1285→  │
  1286→  ▼
  1287→REVIEW_PENDING
  1288→  │
  1289→  ▼
  1290→VALIDATED
  1291→  │
  1292→  ▼
  1293→ELIGIBILITY_READY
  1294→  │
  1295→  ▼
  1296→FINANCIAL_READY
  1297→  │
  1298→  ▼
  1299→DPR_READY
  1300→```
  1301→
  1302→| State | Meaning | Who sets it | Who consumes it |
  1303→|-------|---------|-----------|----------------|
  1304→| `EMPTY` | Project created, no data collected. | App shell (on "New Project") | Interview / Guided Forms |
  1305→| `PARTIAL` | Some fields populated, but Phase 1 (Applicant Discovery) is not yet substantially complete. No eligibility pre-checks possible. | Interview orchestration | Interview (determines which phase to enter) |
  1306→| `DISCOVERING` | Applicant identity and category are known. Interview is in Phases 2–5. Business description, NIC code, and/or financial data may be partially collected. Activity-specific suggestions from the Knowledge Package may be present with source=KNOWLEDGE, verification=UNVERIFIED. | Interview orchestration | Interview, Knowledge Engine (for contextual suggestions) |
  1307→| `COMPLETE` | All interview phases finished. All mandatory fields have values (no MISSING mandatory fields). However, no fields are yet confirmed — all are verification=UNVERIFIED. The profile is *data-complete* but not yet *engine-ready*. | Interview orchestration (Phase 6 Review exit or Phase 7 exit) | Review UI (highlights AI/OCR/KNOWLEDGE-sourced fields), Validation Engine (pre-check) |
  1308→| `REVIEW_PENDING` | All mandatory fields have values and the Validation Engine's pre-check has passed (completeness ≥ threshold, no blocking errors, no contradictions). The profile is displayed as a **project summary** (machinery, costs, employees, loan, subsidy, working capital) and the user must explicitly press **"Confirm Project"** — this action stamps all fields with verification=CONFIRMED. This is the mandatory banking review step: the beneficiary's final acknowledgment that everything is correct. | Validation Engine (pre-check passes) → user action (Confirm Project) | User (project summary screen), Validation Engine (final gate) |
  1309→| `VALIDATED` | The user has pressed "Confirm Project" (all fields stamped verification=CONFIRMED) and the Validation Engine's final gate has passed (Zod, business rules, contradictions). All mandatory fields are now engine-ready (verification=CONFIRMED or VALIDATED). The profile is fully engine-ready. | User action (Confirm Project) + Validation Engine (final gate) | All downstream engines (gate is open) |
  1310→| `ELIGIBILITY_READY` | The Eligibility Engine has run and produced an `EligibilityResult`. This result is snapshotted in the database. The profile is unchanged. | Eligibility Engine (called by feature orchestration) | Financial Engine (may use eligibility context), DPR Engine |
  1311→| `FINANCIAL_READY` | The Financial Engine has run and produced a `FinancialResult`. This result is snapshotted in the database. | Financial Engine (called by feature orchestration) | DPR Engine (requires both eligibility + financials) |
  1312→| `DPR_READY` | The DPR Engine has assembled the document. The PDF Engine has rendered it. The user can view, share, or save the DPR. | DPR Engine + PDF Engine (called by feature orchestration) | User (view, share, save), Import/Export Engine |
  1313→
  1314→### 11.2 Forward transitions
  1315→
  1316→```
  1317→EMPTY ──────────[user starts interview or guided wizard]─────→ PARTIAL
  1318→
  1319→PARTIAL ────────[applicant identity + category known]──────→ DISCOVERING
  1320→
  1321→DISCOVERING ────[all interview phases complete]────────────→ COMPLETE
  1322→
  1323→COMPLETE ───────[Validation Engine pre-check passes
  1324→                 (all mandatory fields have values, no errors)]──→ REVIEW_PENDING
  1325→
  1326→REVIEW_PENDING ──[user presses "Confirm Project" +
  1327→                 Validation Engine final gate passes]─────────→ VALIDATED
  1328→
  1329→VALIDATED ──────[Eligibility Engine runs]──────────────────→ ELIGIBILITY_READY
  1330→
  1331→ELIGIBILITY_READY ──[Financial Engine runs]────────────────→ FINANCIAL_READY
  1332→
  1333→FINANCIAL_READY ──[DPR + PDF Engine runs]──────────────────→ DPR_READY
  1334→```
  1335→
  1336→### 11.3 Edit transitions (going backward)
  1337→
  1338→When the user edits a previously-collected field, the profile may need to move backward:
  1339→
  1340→```
  1341→DPR_READY ──────[user edits any input]─────────────────────→ REVIEW_PENDING
  1342→FINANCIAL_READY ──[user edits financial inputs]────────────→ REVIEW_PENDING
  1343→ELIGIBILITY_READY ──[user edits financial inputs]──────────→ REVIEW_PENDING
  1344→                 (eligibility may still be valid;
  1345→                  financial result is invalidated)
  1346→
  1347→REVIEW_PENDING ──[user edits any field]────────────────────→ COMPLETE
  1348→                 (re-validation required;
  1349→                  downstream engine results are invalidated
  1350→                  but NOT deleted — they become stale snapshots)
  1351→
  1352→VALIDATED ──────[user edits any field]─────────────────────→ REVIEW_PENDING
  1353→                 (re-confirmation required;
  1354→                  downstream engine results are invalidated
  1355→                  but NOT deleted — they become stale snapshots)
  1356→
  1357→ANY pre-validation state ──[user edits a dependency root
  1358→                 (NIC code, sector, category)]────────────→ DISCOVERING
  1359→                 (dependents flagged for re-collection)
  1360→```
  1361→
  1362→**Rules for edit transitions:**
  1363→
  1364→1. **Edits invalidate forward, never backward.** Editing the NIC code (Phase 3 data) invalidates machinery and financials (Phase 4–5 data), but does not invalidate the applicant's name (Phase 1 data).
  1365→2. **Invalidated engine results are not deleted.** They become stale snapshots in the database, tagged with the `knowledgeVersion` and profile version that produced them. This preserves reproducibility and audit history.
  1366→3. **The user must explicitly regenerate.** After an edit, the app informs the user which downstream results are now stale and offers to re-run the affected engines. Already-generated DPRs are never silently rewritten.
  1367→4. **State moves forward one step at a time through the engine pipeline.** The user cannot jump from `DPR_READY` directly to re-running just the Financial Engine — the app walks through `REVIEW_PENDING` → `VALIDATED` → `ELIGIBILITY_READY` → `FINANCIAL_READY`, re-running each engine in sequence. This ensures every gate is re-evaluated.
  1368→
  1369→### 11.4 Resume behavior
  1370→
  1371→When the user opens an existing project:
  1372→
  1373→1. Load the persisted profile from SQLite.
  1374→2. Determine the current state from the stored `status` column and the presence/absence of engine snapshots.
  1375→3. If state is `EMPTY` through `DISCOVERING`: resume the interview at the last active phase.
  1376→4. If state is `COMPLETE`: enter the Review phase (Phase 6) to confirm any non-engine-ready fields, then attempt the validation pre-check.
  1377→5. If state is `REVIEW_PENDING`: present the project summary screen. The user can edit (which moves state backward per §11.3) or press "Confirm Project" to proceed.
  1378→6. If state is `VALIDATED` or later: present the project dashboard showing engine results. The user can edit (which moves the state backward per §11.3) or view/export the DPR.
  1379→7. If the user's edit moves the state backward, re-run the interview/workflow from the appropriate point.
  1380→
  1381→### 11.5 State machine and the validation gate
  1382→
  1383→The state machine **is** the enforcement of Design Principle 14 ("Project Completion & Validation gates the engines").
  1384→
  1385→- States `EMPTY` through `COMPLETE` are **pre-validation**. No engine may run.
  1386→- The transition `COMPLETE → REVIEW_PENDING` requires the Validation Engine's **pre-check** to pass — all mandatory fields must have values (no MISSING fields), completeness must meet the threshold, and no blocking errors or unresolved contradictions. Verification is NOT checked here — that is the purpose of REVIEW_PENDING.
  1387→- `REVIEW_PENDING` is the **user review and confirmation checkpoint**. The app displays a project summary (machinery, project cost, employees, loan amount, subsidy, working capital). When the user presses **"Confirm Project,"** all fields are stamped with verification=CONFIRMED. This creates a clean audit trail: the beneficiary has seen and approved the full project before any calculation runs.
  1388→- The transition `REVIEW_PENDING → VALIDATED` requires the Validation Engine's **final gate** to pass — Zod schema validation, business rules, and contradiction checks run against the now-confirmed profile. This is a lightweight check since data presence and completeness were already verified at the pre-check.
  1389→- States `VALIDATED` through `DPR_READY` are **post-validation**. Engines run sequentially, each producing a snapshot that the next engine consumes.
  1390→
  1391→### 11.6 Persistence
  1392→
  1393→The profile state is stored in the `project` table (doc 03) as the `status` column. The column holds the current state string (`EMPTY`, `PARTIAL`, `DISCOVERING`, `COMPLETE`, `REVIEW_PENDING`, `VALIDATED`, `ELIGIBILITY_READY`, `FINANCIAL_READY`, `DPR_READY`). Engine snapshots are stored in their respective tables (`project_financials`, `dpr_document`) keyed to the project.
  1394→
  1395→The state column is the single source of truth for the UI (which screens to show), the interview (whether to resume or present results), and the feature orchestration (which engines to run or re-run).
  1396→
  1397→### 11.6.1 Extensibility
  1398→
  1399→The state machine is designed to accommodate future post-DPR states without breaking existing transitions. Potential additions (not for v1, but architecturally expected):
  1400→
  1401→- **`BANK_SUBMITTED`** — The DPR has been submitted to a bank or financial institution.
  1402→- **`FINALIZED`** — The project is locked — no further edits permitted. The DPR is the authoritative version.
  1403→
  1404→These would extend the pipeline after `DPR_READY`. Adding them would not change any existing transitions — they would add new forward transitions from `DPR_READY` and new edit transitions (e.g., `BANK_SUBMITTED → REVIEW_PENDING` if the bank requests changes).
  1405→
  1406→The `status` column uses a free-text string (not an enum in SQLite), so new states can be added without a schema migration.
  1407→
  1408→### 11.7 State machine and provenance interaction
  1409→
  1410→The state machine and the provenance model (Section 9) interact at two critical points:
  1411→
  1412→**1. The `COMPLETE → REVIEW_PENDING` transition (data gate):**
  1413→
  1414→- A profile in `COMPLETE` state has all mandatory fields populated (no MISSING fields).
  1415→- The Validation Engine checks completeness, blocking errors, and contradictions.
  1416→- Verification is NOT a factor here — fields are expected to be UNVERIFIED at this point.
  1417→- If any mandatory field is MISSING, or if there are blocking errors/contradictions, the transition is **blocked**.
  1418→
  1419→**2. The `REVIEW_PENDING → VALIDATED` transition (confirmation gate + final validation):**
  1420→
  1421→- The user has pressed "Confirm Project," which stamps all fields with verification=CONFIRMED.
  1422→- The Validation Engine performs the **final gate** check: Zod validation, business rules, contradictions.
  1423→- On success, all field provenance entries have their `verification` stamped to `VALIDATED`.
  1424→
  1425→This ensures the principle is enforced at the architectural level: **no value reaches a deterministic engine without user confirmation.** The REVIEW_PENDING screen is the single, non-bypassable confirmation point for all fields regardless of source.

  1484→## 13. Success Criteria
  1485→
  1486→This architecture document must answer these questions unambiguously:
  1487→
  1488→1. **What exactly is a Structured Project Profile?**
  1489→   → Section 2. A typed, Zod-validated object with 13 top-level sections covering applicant, business, location, land, capacity, machinery, raw materials, employees, utilities, financials, working capital, market, attachments, validation metadata, provenance metadata, and completion metadata. Both the AI interview and guided forms produce an identical instance.
  1490→
  1491→2. **How does the AI decide the next question?**
  1492→   → Section 3, Stages 4–7. The Validation Engine identifies missing fields and contradictions. The dependency graph determines which fields are askable. The phase model determines the current focus area. The question planner selects the highest-priority askable field.
  1493→
  1494→3. **How are NIC activities discovered?**
  1495→   → Section 6. The Knowledge Engine searches the NIC code database using the beneficiary's business description. Candidates are presented for user confirmation. Once confirmed, NIC code, sector, sub-category, and activity type are fixed. PMEGP rules are then determined.
  1496→
  1497→4. **How does the Knowledge Package assist beneficiaries?**
  1498→   → Section 5. When a beneficiary cannot answer a technical question, the interview queries the Knowledge Package for suggested values (machinery, costs, benchmarks). Suggestions are presented for explicit acceptance — never silently applied.
  1499→
  1500→5. **How are suggestions handled?**
  1501→   → Section 10. Knowledge Package → Knowledge Engine → Suggestion → User Review → Acceptance → Profile → Validation → Engines. Suggestions never bypass confirmation. AI-generated suggestions are prohibited.
  1502→
  1503→6. **How are unknown answers handled?**
  1504→   → Section 8. Uncertainty is detected → Knowledge Package provides suggestion → AI explains suggestion → User accepts, edits, or skips. "I don't know" is never an error and never blocks the conversation.
  1505→
  1506→7. **How does provenance work?**
  1507→   → Section 9. Every field carries two orthogonal dimensions: Source (USER, AI, OCR, KNOWLEDGE, null) and Verification (UNVERIFIED, CONFIRMED, VALIDATED). Engine readiness is determined solely by verification — a field must be CONFIRMED or VALIDATED for engines to consume it. Source is for audit trails and DPR annotation, not for gate logic. The REVIEW_PENDING screen is the single mandatory confirmation point for all fields.
  1508→
  1509→8. **How do interview phases work?**
  1510→   → Section 4. Seven sequential phases: Applicant Discovery → Business Discovery → Activity Resolution → Project Sizing → Financial Planning → Review → Validation Completion. Each phase has a clear objective and exit condition. Within a phase, question order is flexible; phase order is fixed by dependencies.
  1511→
  1512→9. **How do deterministic engines receive clean, validated data?**
  1513→   → The Validation Engine (doc 04 §5, doc 15 §2) gates all downstream engines. Engines receive the Structured Project Profile only after validation passes. The profile is the single data contract — engines never see conversation messages, form state, or provenance metadata.
